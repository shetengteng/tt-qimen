# 后续扩展规划（Post-MVP）

> **所属文档**：方案设计 · 子模块  
> **版本**：v1.0  
> **前置文档**：可行性分析 §5 "其余 12 种方法可行性等级"

## 1. 目标

MVP 上线并稳定运行 30 天后，根据 **留存数据 + 用户反馈** 决定是否扩展。本文件登记所有"候选扩展"项目，方便未来快速决策。

## 2. 候选扩展列表（按优先级）

### P0 - 强推荐扩展（预计第 2-3 月）

#### 2.1 塔罗牌（Tarot）

- **定位**：西方命理，年轻女性主力用户群
- **技术难度**：★★☆☆☆
- **数据需求**：78 张牌面（正/逆位含义）+ 3 张/5 张牌阵
- **核心依赖**：图片资源（可用公版 Rider-Waite）
- **预估工时**：4-5 天
- **关键文件**：
  - `src/pages/TarotView.vue`
  - `src/data/tarot/cards.ts`（78 × 2 = 156 段含义）
  - `src/data/tarot/spreads.ts`（牌阵模板）
  - `public/images/tarot/*.svg`
- **风险**：图片版权需严格核查（Rider-Waite 在美国 1971 年进入公版，中国状态待确认）

#### 2.2 八字 - 分面深度分析

- **定位**：八字增强，按事业/感情/财富/健康分面给模板化解读
- **技术难度**：★★★☆☆
- **数据需求**：`4 分面 × 10 十神 × 3 吉凶 = 120 段` 模板
- **核心依赖**：复用 MVP 八字算法
- **预估工时**：3-4 天（数据 2 天 + UI 1-2 天）
- **关键文件**：
  - `src/pages/bazi/components/AspectAnalysis.vue`
  - `src/data/bazi/aspectHints.ts`
- **风险**：分面维度爆炸风险，需严格控制模板粒度

### P1 - 推荐扩展（预计第 4-6 月）

#### 2.3 奇门遁甲简化版

- **定位**：专业向"神秘感"加强，八字用户的进阶选择
- **技术难度**：★★★★☆
- **数据需求**：九宫八卦基础 + 时家奇门起局表
- **核心依赖**：`tyme4ts` 已有时辰排盘，需额外写"九星/八门/八神"排列算法
- **预估工时**：7-10 天
- **关键文件**：
  - `src/pages/QimenView.vue`
  - `src/core/qimen/chart.ts`
  - `src/data/qimen/*.ts`
- **风险**：**派别繁多**（转盘派 vs 飞盘派），需明确选一派

#### 2.4 生肖配对

- **定位**：轻量社交娱乐
- **技术难度**：★☆☆☆☆
- **数据需求**：`12 × 12 = 144` 段组合解读
- **核心依赖**：纯自研
- **预估工时**：2 天
- **特点**：适合作为"日常小工具"入口，留存价值一般

#### 2.5 星座运势

- **定位**：年轻用户入口
- **技术难度**：★★☆☆☆
- **数据需求**：12 星座 × 7 日运势（需每日更新）
- **核心依赖**：**需外部 API 或每日 LLM 生成**（与"静态网站"定位冲突，需评估）
- **预估工时**：3-5 天 + 日常运维
- **风险**：**运维成本高**（每日生成/采集），违反静态站原则

### P2 - 试验性扩展（看情况）

- **定位**：强趣味，AI 卖点
- **技术难度**：★★★★★

## 4. 技术债务扩展

到 Post-MVP 阶段，以下技术债务可能需要偿还：

### 4.1 数据层迁移

**问题**：所有数据目前以 TypeScript 文件内嵌，`content.md` 手改成本高  
**方案**：迁移到 Contentlayer / Nuxt Content / 自建 JSON → TS 编译脚本

### 4.2 内容管理后台

**问题**：C 类文案需要内容编辑频繁更新  
**方案**：Strapi / Directus / TinaCMS（headless CMS 方案）

### 4.3 用户系统

**问题**：生辰信息目前只存 localStorage，无法跨设备同步  
**方案**：最小化用户系统（邮箱/微信 OAuth + Firebase/Supabase 作持久化）

### 4.4 国际化（i18n）

**问题**：中文内容，但海外华人/东南亚有需求  
**方案**：vue-i18n + 简繁/英文版本

### 4.5 PWA

**问题**：用户希望离线可用（尤其八字排盘）  
**方案**：`vite-plugin-pwa` + Service Worker 缓存静态资源和核心数据

## 5. 决策门槛

每个 Post-MVP 扩展开启前，需满足以下**至少 2 条**：

- [ ] MVP 月活（MAU）超过 10000
- [ ] 目标模块在用户反馈中"想要"占比 > 30%
- [ ] 核心指标（留存、次日回访）达到既定目标
- [ ] 有稳定的内容审校团队（> 1 人）
- [ ] 对应的商业化模型已验证
