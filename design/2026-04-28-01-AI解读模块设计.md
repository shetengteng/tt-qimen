## 1. 概述与目标

### 1.1 背景

当前每个模块的"分享卡 / 解读"是 **模板化文本**（`InterpretBlock` / `interpretTemplate.ts`），优点是确定性、零依赖；缺点是颗粒度粗、无法回答"我今年事业怎么样"这类自然语言问题。

引入 AI 解读模块，让用户在排盘完成后能：

- 一键看到针对**当前命盘**的更长、更具体的 AI 解读；
- 围绕该命盘做**多轮自由提问**；
- 对话与排盘**绑定**：换一组生辰 = 新会话，避免上下文污染。

### 1.2 目标

| # | 目标 |
|---|---|
| O1 | 在排盘结果区"分享"按钮后增加 **询问 AI** 按钮，吊起右侧 Drawer |
| O2 | Drawer 内：欢迎语 → 自动首次解读 → 多轮对话；流式输出 |
| O3 | API Key / Model / Temperature 等配置在 **`/settings`** 页面统一管理 |
| O4 | MVP 仅 DeepSeek，但代码层抽象 `LlmProvider` 接口，后续可扩 OpenAI / Claude |
| O5 | 上下文 = 模块结构化 JSON（**信息尽可能丰富**） + system prompt + 模块化预设问题 |
| O6 | 8 个模块全覆盖，每模块自带 4-6 个预设快捷提问 |
| O7 | 对话**按命盘指纹持久化**到 localStorage：换一个生辰自动切到新会话 |
| O8 | 隐私：API Key 仅存浏览器、所有请求浏览器直发 LLM、不经任何第三方后端 |

### 1.3 评估指标

- **首次解读 TTFB**：< 2s（取决于 DeepSeek 网络）
- **bundle 增量**：≤ 30KB gzipped（动态 import 隔离）
- **首屏 LCP 不退化**：AI 模块代码不进首屏
- **8 模块复用率**：1 套 Drawer 组件 + 8 套上下文构造器

## 3. 用户场景

### 3.1 主场景：八字命盘 + 询问 AI

```
1. 用户在 /bazi 输入生辰 → 点 "排盘" → 看到四柱 / 大运 / 流年
2. 滚动到底部，看到 [分享] [询问 AI] [重新排盘] 三个按钮
3. 点 "询问 AI"：
   - 若未配置 API Key → Drawer 内提示 "请先在 设置 中配置 API Key"，
     带 [前往设置] 链接（跳 /settings）
   - 若已配置 → Drawer 滑入，自动发起 "首次解读" 流式回复
4. 用户看到首次解读后，可：
   - 点击预设提问（如 "今年事业运势"）一键发问
   - 自由输入问题
5. 关闭 Drawer，再次点 "询问 AI" → 同一命盘对话历史复现
6. 改一个生辰重新排盘 → "询问 AI" 进入新会话（旧会话静默归档）
```

### 3.2 次场景：跨模块切换

```
- 在 /bazi 跟 AI 聊到一半，切到 /ziwei
- 紫微页打开 "询问 AI" → 看到的是紫微的会话（不混淆）
- 回到 /bazi → 八字会话原样恢复
```

### 3.3 边缘场景

- 网络中断 / Key 失效 → 错误气泡 + 重试按钮
- 用户主动 Cancel 流式输出 → 保留已生成的部分作为助手消息
- localStorage 配额满 → 自动按 LRU 删除最旧会话

## 4. 关键设计决策（已确认）

| 决策项 | 选项 | 备注 |
|---|---|---|
| UI 形式 | **右侧抽屉 Drawer** | 桌面 480px，移动端全屏 |
| 上下文 | **结构化 JSON，尽可能丰富** | 不抓 DOM 文本，从 chart 对象抽取 |
| 流式 | **是** | DeepSeek SSE，**复用 OpenAI 官方 SDK**（不造轮子） |
| 历史 | **按命盘指纹持久化** | fingerprint = sha1(moduleId + 关键入参) |
| Key 入口 | **独立设置页 `/settings`** | AI 配置作为其中一节 |
| Provider | **MVP 仅 DeepSeek，代码预留接口** | `LlmProvider` 接口；DeepSeek 通过 OpenAI SDK + baseURL 接入 |
| 预设提问 | **是，每模块 4-6 个** | 国际化 + 模块定制 |
| 模块范围 | **全部 8 个** | 但实施分两个 wave |
| LLM 客户端框架 | **`openai` 官方 npm 包** | DeepSeek 官方文档背书；流式/abort/错误模型/重试全自带 |
| 长会话渲染 | **分层 + 虚拟列表** | 锚定区走普通 DOM，历史区超阈值后用 `vue-virtual-scroller`；详见 §13 |
| 流式排版引擎 | **Pretext.js** | 用户主动指引；2026 新爆款（43K stars）；Canvas 测高 0 reflow，60-120fps；专门解决"流式 token 抖动"P3 痛点 |

## 5. 整体架构

### 5.1 分层

```
                 ┌────────────────────────────────────────────────┐
   UI 层         │  AskAiButton  ·  AiInterpretDrawer  ·  /settings│
                 └─────────────────┬──────────────────────────────┘
                                   │
                 ┌─────────────────▼──────────────────────────────┐
   Composables   │  useAiChat       useChatHistory   useAiConfig   │
                 │  (流式 / 取消)   (按指纹归档)     (Key/Model)   │
                 └────────┬─────────────────┬───────────────┬─────┘
                          │                 │               │
                 ┌────────▼─────────┐ ┌─────▼───────┐ ┌─────▼──────┐
   Domain        │ contextBuilders/ │ aiHistory    │ aiConfig     │
                 │ (8 个模块)       │ Pinia store  │ Pinia store  │
                 └────────┬─────────┘ └──────┬──────┘ └──────┬─────┘
                          │                  │               │
                 ┌────────▼──────────────────▼───────────────▼─────┐
   Infra         │ LlmProvider (接口) → DeepSeekProvider (impl)   │
                 │ + storageAdapter (localStorage with quota mgr)  │
                 └─────────────────────────────────────────────────┘
```

### 5.2 数据流（一次首次解读）

```
[用户点 询问 AI]
    │
    ▼
AskAiButton 打开 Drawer (传 moduleId + chart ref)
    │
    ▼
AiInterpretDrawer onOpen():
    1. 计算 fingerprint(moduleId, birthInputs)
    2. useChatHistory.loadOrCreate(fingerprint)
    3. 若是新会话:
         a. contextBuilders[moduleId].build(chart) → 结构化 context
         b. 构造首条 system + user message（首次解读）
         c. useAiChat.send() 触发流式
    4. 否则: 渲染历史，等待用户输入
    │
    ▼
useAiChat.send(messages):
    - 调 LlmProvider.streamChat(messages) → AsyncIterable<chunk>
    - 每个 chunk 推到 reactive history
    - 完成后落盘到 useChatHistory
    │
    ▼
DeepSeekProvider.streamChat:
    - fetch(POST https://api.deepseek.com/v1/chat/completions, stream=true)
    - 解析 SSE → yield delta.content
```

## 7. 上下文构造器（核心）

### 7.1 设计哲学

不抓 DOM 文本，因为：

- DOM 包含大量 UI 装饰文字（"展开 / 收起"、"◎"），噪声大
- 截图相同信息走 chart JSON 更紧凑、可控
- 8 个模块各自的 `chart` 对象已经是计算结果的"权威源"

但用户要求 **信息尽可能丰富**，所以构造器做两件事：

1. **JSON 摘要**（机器可读，结构化字段）
2. **自然语言陈述**（生成 200-400 字的 Markdown 描述，便于 LLM 直接引用）

### 7.2 接口

```ts
// src/composables/ai/contextBuilders/types.ts
export interface AiContext {
  /** 用于 fingerprint：moduleId + 关键入参 */
  fingerprint: string

  /** 模块标识（影响 system prompt 选择） */
  moduleId: ModuleId

  /** 用户友好的命盘标识，显示在 Drawer 头部 */
  displayLabel: string  // 如 "1990-05-20 午 · 男"

  /** 给 LLM 的自然语言陈述（Markdown，已含分段） */
  narrative: string

  /** 给 LLM 的结构化字段（JSON 字符串，附在 narrative 之后） */
  structured: Record<string, unknown>

  /** 模块定制的 4-6 个预设提问 i18n key */
  presetPromptKeys: string[]
}

export interface ContextBuilder<TChart> {
  /** 从 chart + birth 输入构造完整 AiContext */
  build(input: { chart: TChart; locale: Locale; t: I18nT }): AiContext
}
```

## 8. LLM 客户端（流式）

### 8.1 选型原则：复用 OpenAI 官方 SDK，不造轮子

DeepSeek 官方文档（2026-04 最新）明确：

> The DeepSeek API uses an API format compatible with OpenAI/Anthropic. By modifying the configuration, you can use the OpenAI SDK to access the DeepSeek API.

因此 LLM 调用层 **直接用 `openai` 官方 npm 包**：

| 维度 | 评估 |
|---|---|
| 行业地位 | 18.9M 周下载，事实标准 |
| 浏览器支持 | 显式支持 `dangerouslyAllowBrowser: true`（用户 Key 自管） |
| baseURL 切换 | 一行配置即可指向 DeepSeek / OpenAI / 自建代理 |
| 流式 API | 原生 `AsyncIterable`，`for await` 直接消费 |
| AbortController | SDK 内置 `signal` 透传 |
| 错误模型 | `APIError`、`AuthenticationError`、`RateLimitError` 等子类，便于映射 i18n |
| 重试 / 退避 | SDK 内置 `maxRetries`，无需手写 |
| 体积 | ~152KB（动态 import 隔离，不进首屏） |

**否决原生 fetch 方案**：用户明确要求"复用已有框架，不造轮子"；自写 SSE 解析、错误映射、abort、重试都是已经被官方 SDK 解决的问题。

**否决 Vercel AI SDK**：`streamText` + `useChat` 的核心抽象基于 server-side route handler；浏览器端用法非主流且文档少；本项目无后端的形态下没有杠杆。

### 8.3 DeepSeek 实现（基于 OpenAI SDK）

```ts
// src/composables/ai/providers/deepseek.ts
import OpenAI from 'openai'
import { APIError } from 'openai'
import type { LlmProvider } from './types'
import type { AiConfig } from '../types'
import { LlmError } from '../errors'

function buildClient(config: AiConfig) {
  return new OpenAI({
    apiKey: config.apiKey,
    baseURL: config.baseUrl,        // 默认 https://api.deepseek.com
    dangerouslyAllowBrowser: true,  // 项目纯静态 SPA，Key 自管
    maxRetries: 1,                  // 流式不便重试，留 1 次给非流式 ping
  })
}

export const deepseekProvider: LlmProvider = {
  id: 'deepseek',

  async *streamChat(messages, config, options) {
    const client = buildClient(config)
    try {
      const stream = await client.chat.completions.create(
        {
          model: config.model,
          messages,
          temperature: config.temperature,
          stream: true,
        },
        { signal: options?.signal },
      )
      for await (const chunk of stream) {
        const delta = chunk.choices[0]?.delta?.content
        if (delta) yield delta
      }
    } catch (e) {
      throw mapError(e)
    }
  },

  async ping(config) {
    const client = buildClient(config)
    try {
      await client.chat.completions.create({
        model: config.model,
        messages: [{ role: 'user', content: 'ping' }],
        max_tokens: 1,
      })
      return { ok: true }
    } catch (e) {
      const err = mapError(e)
      return { ok: false, message: err.detail }
    }
  },
}

function mapError(e: unknown): LlmError {
  if (e instanceof APIError) return new LlmError(e.status ?? 0, e.message, e.code ?? undefined)
  if ((e as Error).name === 'AbortError') return new LlmError(0, 'aborted', 'aborted')
  return new LlmError(0, String(e))
}
```

**关键要点**：

1. `for await (const chunk of stream)` —— SDK 已经封装好 SSE 解析、`[DONE]` 终止、keep-alive 处理
2. `signal` 直接透传到 fetch，`controller.abort()` 即可中止流式
3. `APIError` 子类映射到自家 `LlmError`，state code 一一对应（401 / 429 / 500 / network）
4. `dangerouslyAllowBrowser: true` 是 SDK 强制要求；隐私页明示后用户已知风险

### 8.4 useAiChat composable

```ts
// src/composables/ai/useAiChat.ts
export function useAiChat(provider: Ref<LlmProvider>, config: Ref<AiConfig>) {
  const messages = ref<ChatMessage[]>([])
  const streaming = ref(false)
  const error = ref<LlmError | null>(null)
  let abortController: AbortController | null = null

  async function send(input: string, contextMessages: ChatMessage[] = []) {
    error.value = null
    const userMsg: ChatMessage = { role: 'user', content: input }
    messages.value.push(userMsg, { role: 'assistant', content: '' })

    streaming.value = true
    abortController = new AbortController()
    try {
      const stream = provider.value.streamChat(
        [...contextMessages, ...messages.value.slice(0, -1)],  // 排除占位的 assistant
        config.value,
        { signal: abortController.signal },
      )
      for await (const chunk of stream) {
        const last = messages.value[messages.value.length - 1]
        last.content += chunk
      }
    } catch (e) {
      if ((e as Error).name === 'AbortError') {
        // 用户主动停止，保留已生成内容
      } else {
        error.value = e instanceof LlmError ? e : new LlmError(0, String(e))
      }
    } finally {
      streaming.value = false
      abortController = null
    }
  }

  function stop() {
    abortController?.abort()
  }

  function reset() {
    stop()
    messages.value = []
  }

  return { messages, streaming, error, send, stop, reset }
}
```

### 11.2 命盘指纹算法

```ts
// 用 Web Crypto API（浏览器原生），不引依赖
async function buildFingerprint(moduleId: ModuleId, params: Record<string, unknown>): Promise<string> {
  const sorted = Object.keys(params).sort().reduce((acc, k) => ({ ...acc, [k]: params[k] }), {})
  const text = `${moduleId}|${JSON.stringify(sorted)}`
  const buf = await crypto.subtle.digest('SHA-1', new TextEncoder().encode(text))
  return Array.from(new Uint8Array(buf)).map(b => b.toString(16).padStart(2, '0')).join('').slice(0, 16)
}
```

## 12. 开源库选型与对比

### 12.1 LLM 调用层

| 方案 | 适用 | 评分 | 备注 |
|---|---|---|---|
| **`openai` 官方 SDK** | ✅ 选用 | ★★★★★ | DeepSeek 官方文档亲自背书；18.9M 周下载；流式 / abort / 错误模型 / 重试全自带 |
| 原生 fetch + ReadableStream | ❌ 不用 | ★★★★ | 自己造轮子（SSE 解析 / 错误映射 / abort），违背 "复用已有框架" 原则 |
| Vercel AI SDK (`ai` + `@ai-sdk/vue`) | ❌ 不用 | ★★ | `useChat` 强依赖 server-side `/api/chat`，本项目无后端 |
| `openai-fetch` (社区) | ❌ 不用 | ★★★ | 14KB 更轻量，但维护活跃度不如官方，错误模型不全 |
| `eventsource-parser` | ❌ 不用 | ★★★ | 仅 SSE 解析工具，仍需自己拼装客户端，半成品 |

**结论**：直接用 `openai` 官方 SDK，配合 `baseURL: 'https://api.deepseek.com'` 接 DeepSeek。

**体积权衡**：SDK ~152KB（gzip 后约 50KB）但通过 **动态 import** 隔离到 AI chunk，不进首屏 bundle，不影响排盘 LCP。用户首次点 "询问 AI" 时按需加载，符合用户感知模型。

**未来扩展 OpenAI / 其他 OpenAI 兼容 Provider**：换一个 baseURL + apiKey 即可，0 代码改动。Anthropic 走 `@anthropic-ai/sdk`，新增 `anthropicProvider.ts`，与 `LlmProvider` 接口一致。

### 12.2 Markdown 流式渲染

| 方案 | 评分 | 备注 |
|---|---|---|
| **markstream-vue** | ★★★★★ | 专为 AI 流式优化，11K 周下载，零抖动，progressive；自带 **虚拟窗口模式** + 增量解析；推荐 `parseMarkdownToStructure` + `<MarkStream :nodes :final />` 模式（不重复解析） |
| streamdown-vue | ★★★★ | 类似 markstream-vue，社区稍小 |
| vue-stream-markdown | ★★★★ | 增量节点补全防抖 + 渐进式 Mermaid，新兴方案 |

**结论**：选 **markstream-vue**。除了流式渲染外，还自带"虚拟窗口"应对长 markdown，与下方虚拟列表分工互补：

- **markstream-vue 虚拟窗口** = 单条消息内 markdown 太长时，DOM 节点虚拟化（如长篇综合解读）
- **vue-virtual-scroller** = 消息条数过多时，整个消息列表虚拟化

|---|---|---|

### 12.5 文本排版引擎（针对流式高度抖动）

| 方案 | 评分 | 备注 |
|---|---|---|
| **Pretext.js**（`@chenglou/pretext`） | ★★★★★ | Cheng Lou（前 React 核心成员）2026/04 发布，24 小时 16K stars 现 43K+；**框架无关**（React/Vue/Svelte/vanilla）；15KB；用 Canvas `measureText` 绕过 DOM reflow；**官方 demo 含 dynamic AI chat bubbles + variable-height virtual scroll**；多语言（CJK/Arabic）支持 |
| 浏览器原生 ResizeObserver | ★★★★ | vue-virtual-scroller 内置；触发 reflow，流式中频繁触发会抖 |
| 手写 getBoundingClientRect | ★★ | 同样 reflow，更慢 |

**Pretext.js 在本项目的定位**：**不替代** vue-virtual-scroller，而是作为"流式排版预测器"切入最难的 P3 痛点（流式中高度持续变化）。

**核心 API**：

```ts
import { prepare } from 'pretext'

// 一次准备（按主题字体配置）
const ctx = prepare({ fontFamily: 'Inter', fontSize: 16, lineHeight: 1.6 })

// 多次 layout：每次仅 ~0.09ms（500 块文本）
const { height, lines } = ctx.layout(message.content, { width: 432 /* drawer 内宽 */ })
```

**收益**：

- 流式 token 来时：用 Pretext 预测当前内容**至少**需要的高度，给消息气泡设 `min-height`
- markdown 实际渲染高度始终 ≥ Pretext 估算（标题 / 列表 / 代码块带固定额外间距）
- 虚拟列表的滚动条不再因为流式 token 抖动 —— 因为容器 min-height 提前定死
- 0 reflow，60-120fps 平滑

**注意**：Pretext 测的是 plain text；markdown 含 `**粗体**`、`# 标题`、列表等会让真实高度高于估算，但**这恰好是我们想要的**——估算值作为下界，配合 `min-height` 防抖。详细策略见 §13.4。

## 13. 渲染性能策略（虚拟列表 + 分层渲染）

> 用户明确要求：对话内容多时渲染会有问题，需要单独优化。
> 本节系统性回答"长会话怎么不卡"。

### 13.1 性能挑战的来源

聊天场景的真正痛点不是单纯的"DOM 节点多"，而是 **三个叠加的性能压力**：

| 压力源 | 触发条件 | 表现 |
|---|---|---|
| P1 消息条数多 | > 50 条消息 | 滚动时整个 list re-render，掉帧 |
| P2 单条 markdown 长 | 单条 > 2000 字 | 解析成本高，每个 token 流式过来都重解析 |
| P3 流式末尾高度持续变化 | 流式中 | 虚拟列表测高 cache 失效，引起 list reflow |

4 层组合方案 **各打一拳**：

| 层级 | 工具 | 解决的压力 |
|---|---|---|
| 列表虚拟化层 | **vue-virtual-scroller** | P1 消息条数多 |
| 单条文档虚拟化 | **markstream-vue 虚拟窗口** | P2 单条 markdown 长 |
| 流式分层渲染 | **HISTORY/ANCHOR 自家方案** | P3 流式末尾测高失效 |
| 文本排版预测 | **Pretext.js** | P3 进一步根除（min-height 提前定死） |

### 13.2 核心策略：分层渲染（Layered Rendering）

```
┌──────────────────────────────────────────────────────────┐
│   AI Drawer 内的消息区分两个层（Z 轴上是相邻区域）       │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │  HISTORY ZONE  （历史区，可虚拟化）                │  │
│  │  ─────────────────────────────────────────         │  │
│  │  · messages[0..N-2] 全部走 vue-virtual-scroller    │  │
│  │  · 阈值切换：messages.length ≤ 30 时全渲染，       │  │
│  │    > 30 时启用 DynamicScroller                     │  │
│  │  · 每条已完成消息 final=true 一次性解析 markdown   │  │
│  │  · ResizeObserver 上报高度给 DynamicScroller       │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │  ANCHOR ZONE   （锚定区，永远普通 DOM）            │  │
│  │  ─────────────────────────────────────────         │  │
│  │  · messages[N-1]（当前正在生成 / 最近一条）        │  │
│  │  · 流式中：final=false，markstream-vue 增量渲染    │  │
│  │  · 流式完成 → 该消息"沉降"到 HISTORY ZONE         │  │
│  │  · 高度变化不影响虚拟列表的 height cache           │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  滚动条横跨 HISTORY + ANCHOR，逻辑上是一个流              │
└──────────────────────────────────────────────────────────┘
```

**沉降时机**：流式 done → `useAiChat` 把最后一条消息标记 `final=true` → 模板的 `v-if` 切换到 HISTORY ZONE。这一步：

- 不改变滚动位置（沉降前后该消息屏幕坐标一致）
- 不重解析 markdown（markstream-vue 沿用已有 nodes，加一个 `final` 属性）
- DynamicScroller 收到新条目后通过 ResizeObserver 自动测高

### 13.3 模板伪代码

```vue
<!-- AiInterpretDrawer.vue -->
<template>
  <div class="msg-list">
    <!-- HISTORY ZONE：阈值切换 -->
    <template v-if="historyMessages.length <= VIRTUALIZE_THRESHOLD">
      <AiMessageBubble
        v-for="m in historyMessages"
        :key="m.id"
        :message="m"
        final
      />
    </template>
    <DynamicScroller
      v-else
      :items="historyMessages"
      :min-item-size="80"
      key-field="id"
      class="msg-virtual-list"
    >
      <template #default="{ item, active }">
        <DynamicScrollerItem
          :item="item"
          :active="active"
          :data-index="item.id"
        >
          <AiMessageBubble :message="item" final />
        </DynamicScrollerItem>
      </template>
    </DynamicScroller>

    <!-- ANCHOR ZONE：永远普通 DOM -->
    <AiMessageBubble
      v-if="anchorMessage"
      :key="anchorMessage.id"
      :message="anchorMessage"
      :final="!streaming"
      ref="anchorEl"
    />
  </div>
</template>

<script setup lang="ts">
const VIRTUALIZE_THRESHOLD = 30  // 消息数 ≤ 30 全渲染，> 30 启用虚拟化

const historyMessages = computed(() =>
  streaming.value ? messages.value.slice(0, -1) : messages.value.slice(0, -1)
)
const anchorMessage = computed(() =>
  streaming.value ? messages.value[messages.value.length - 1] : null
)
</script>
```

> 注：当不是流式中时（`streaming === false`），`anchorMessage` 为 null，所有消息都进 HISTORY ZONE。流式开始后立即 push 一条占位 assistant 消息进 ANCHOR ZONE，流式结束后整体重计算时该消息自然落入 HISTORY。

### 13.4 Pretext.js 流式高度预测（根除 P3）

`vue-virtual-scroller` + 分层渲染已能 90% 解决问题，但 ANCHOR ZONE 内的流式消息仍然会因为 token 持续追加，让父容器高度抽搐式变化（虽然不影响 HISTORY ZONE 的虚拟列表）。

**Pretext.js 的接入方式**：在 `AiMessageBubble` 流式中用 Pretext 预测高度，给容器写 `min-height`。markdown 实际渲染会让真实高度 ≥ 预测值，配合 CSS `min-height` 始终向下不向上跳，杜绝抖动。

```ts
// composables/ai/usePretextPredictor.ts
import { prepare } from '@chenglou/pretext'
import { computed, ref, watchEffect, type Ref } from 'vue'

let cachedCtx: ReturnType<typeof prepare> | null = null

function getCtx(theme: 'guofeng' | 'minimal') {
  // 不同主题字体不同；缓存 prepare 结果
  if (cachedCtx) return cachedCtx
  cachedCtx = prepare({
    fontFamily: theme === 'guofeng'
      ? '"Noto Serif SC", "STSong", serif'
      : '"Inter", "PingFang SC", sans-serif',
    fontSize: 14,
    lineHeight: 1.7,
  })
  return cachedCtx
}

export function usePretextPredictor(
  content: Ref<string>,
  containerWidth: Ref<number>,  // 由 ResizeObserver 监听 drawer 内宽
  theme: Ref<'guofeng' | 'minimal'>,
) {
  const minHeight = ref(0)

  watchEffect(() => {
    if (!content.value || !containerWidth.value) return
    const ctx = getCtx(theme.value)
    const { height } = ctx.layout(content.value, { width: containerWidth.value })
    // markdown 标题 / 列表 / 代码块会让真实高度更高，给 1.1 系数留余量
    minHeight.value = Math.ceil(height * 1.0)
  })

  return minHeight
}
```

```vue
<!-- AiMessageBubble.vue -->
<template>
  <div
    class="bubble"
    :class="message.role"
    :style="{ minHeight: !final ? `${pretextMinHeight}px` : undefined }"
  >
    <MarkStream :nodes="nodes" :final="final" />
    <span v-if="!final && isStreaming" class="cursor">▌</span>
  </div>
</template>

<script setup lang="ts">
const pretextMinHeight = usePretextPredictor(
  toRef(props.message, 'content'),
  drawerInnerWidth,  // 来自 inject 的 drawer 内宽
  themeRef,
)
</script>
```

**关键点**：

- `min-height` 仅在 `final=false` 时启用；流式完成后清除，让真实高度生效
- markdown 的 \*\*粗体\*\* / `代码` 等不会让宽度变化，但 `# 标题` 和 `- 列表` 会增加纵向间距（因 line-height + margin），所以预测值是"下界"
- Pretext 的 `prepare` 上下文按主题缓存，切换主题时会自然失效重建
- 测量成本：单次 `~0.09ms`（500 块文本基准），流式中每 token 触发也不卡

**性能基准**（要求达到的）：

- 1000 字消息从 0 流到完整：抽屉滚动条偏移 < 4px
- 5000 字消息从 0 流到完整：min-height 误差 < 5%

### 13.5 markstream-vue 的"增量解析"配合

为避免"每个 token 来一次都重解析整段 markdown"，按 markstream-vue 推荐姿势：

```ts
// composables/ai/useStreamingMarkdown.ts
import { parseMarkdownToStructure } from 'markstream-vue'

export function useStreamingMarkdown(content: Ref<string>, final: Ref<boolean>) {
  // 流式中：使用 throttled 解析（每 50ms 一次）避免每个 token 都重解析
  // final=true 时：一次解析后冻结
  const nodes = ref<MarkdownNode[]>([])
  const parseFn = throttle(() => {
    nodes.value = parseMarkdownToStructure(content.value)
  }, 50)

  watch([content, final], () => {
    if (final.value) {
      nodes.value = parseMarkdownToStructure(content.value)  // 立即解析
    } else {
      parseFn()  // 节流解析
    }
  }, { immediate: true })

  return nodes
}
```

```vue
<!-- AiMessageBubble.vue -->
<template>
  <div class="bubble" :class="message.role">
    <MarkStream :nodes="nodes" :final="final" />
    <span v-if="!final && isStreaming" class="cursor">▌</span>
  </div>
</template>
```

**收益**：

- 流式中每秒约 20 token → 全文 parse ≤ 20 次/秒（节流后）
- 完整 nodes 树只有变更部分的子树重渲染（markstream-vue 内部 diff）
- 单条消息超 5000 字时，markstream-vue 自动开"虚拟窗口"模式，仅渲染可视段落

### 13.6 自动滚动（Sticky Bottom）

聊天 UX 必备：

- 用户在最底部时，新消息追加自动 `scrollTo(bottom)`
- 用户上滚阅读历史时，**不打断**，仅显示"⬇ 新消息"按钮跳转

```ts
// composables/ai/useStickyBottom.ts
export function useStickyBottom(scrollEl: Ref<HTMLElement | null>) {
  const atBottom = ref(true)
  const hasNew = ref(false)

  function onScroll() {
    if (!scrollEl.value) return
    const { scrollTop, scrollHeight, clientHeight } = scrollEl.value
    atBottom.value = scrollHeight - scrollTop - clientHeight < 32  // 32px 容差
    if (atBottom.value) hasNew.value = false
  }

  function announceNew() {
    if (atBottom.value) {
      nextTick(() => scrollToBottom())
    } else {
      hasNew.value = true
    }
  }

  function scrollToBottom() {
    if (!scrollEl.value) return
    scrollEl.value.scrollTo({ top: scrollEl.value.scrollHeight, behavior: 'smooth' })
    hasNew.value = false
  }

  return { atBottom, hasNew, onScroll, announceNew, scrollToBottom }
}
```

`announceNew()` 在 `useAiChat.send()` 触发新 message 时调用；流式 token 来时**不调**，避免每 token 都强制滚动。改用 `requestAnimationFrame` 在每次 `chunk` 来后判断 `if (atBottom)` 再滚一次，60fps 流畅。

### 13.7 消息上限策略（防 OOM + 防 token 超）

```ts
const MAX_MESSAGES_PER_SESSION = 50   // 25 轮对话上限
const MAX_CONTEXT_MESSAGES = 20       // 实际发给 LLM 的消息上限

function buildContextWindow(allMessages: ChatMessage[], systemPrompt: string): ChatMessage[] {
  const recent = allMessages.slice(-MAX_CONTEXT_MESSAGES)
  // 始终保留 system + 第一条首次解读（assistant），让上下文锚定
  const firstInterpretation = allMessages.find(m => m.role === 'assistant')
  const head: ChatMessage[] = [{ role: 'system', content: systemPrompt }]
  if (firstInterpretation && !recent.includes(firstInterpretation)) {
    head.push(firstInterpretation)
  }
  return [...head, ...recent]
}
```

显示层：UI 仅展示最后 `MAX_MESSAGES_PER_SESSION` 条；超出时折叠为"已隐藏 N 条早期对话 [展开]"提示。

## 18. 风险与权衡

| 风险 | 等级 | 缓解 |
|---|---|---|
| API Key 在 localStorage 易被窃取（XSS） | 中 | 项目纯静态，无第三方注入；CSP 头由托管平台保证；隐私页明示 |
| DeepSeek 后续封 CORS / 改协议 | 低 | `LlmProvider` 抽象到位，可秒切 OpenAI |
| 长会话 prompt 累积超 token 上限 | 中 | 加最大消息条数（默认 20）+ 自动丢最早消息（保留 system + 首次解读） |
| html2canvas 截图与 AI Drawer 同时打开导致冲突 | 低 | Drawer 位置 fixed top-right，z-index 高于 share-card；截图走 `data-sharing-snapshot` 属性已隔离 |
| 多模块 contextBuilder 维护成本 | 中 | 抽接口 + 8 个独立文件，互不影响；每个模块的命盘类型已稳定 |
| markstream-vue 与 i18n 冲突（流式中换语言） | 低 | 切语言时 reset 当前会话即可；预期低频 |
| 用户切换命盘前未关闭 Drawer | 低 | Drawer 监听 fingerprint 变化，自动 loadOrCreate 新会话 |
| 中文+繁体的 system prompt 语种漂移 | 低 | system prompt 显式包含 `请使用 ${locale} 回答`，且每次发送都注入当前 locale |
| 长会话渲染卡顿 | 中 | §13 分层渲染 + 阈值切换虚拟列表 + markstream 增量解析；4 层叠加防御 |
| 流式中虚拟列表测高跳跃 | 低（已用 Pretext 预测）| 流式消息走 ANCHOR ZONE 不进虚拟列表 + Pretext.js 提前算 min-height；`overflow-anchor: none` 兜底 |
| 用户上滚阅读时被新消息打断 | 低 | useStickyBottom：仅在 `atBottom === true` 时自动滚；上滚时显示"⬇ 新消息"按钮 |
| Pretext.js 测量与真实 markdown 高度差距过大 | 低 | 仅当 `final=false` 用 min-height（始终向下放，不裁剪）；切换主题字体重建 ctx；性能基准 §13.4 |
| Pretext.js 是新库（2026-04），生态可能未稳 | 低 | API 极简（`prepare` + `layout` 共两个），即使停更也容易自维护；MIT；43K stars 社区健康 |

## 附录 A：DeepSeek API 备忘（2026-04 最新）

- Base URL：`https://api.deepseek.com`（OpenAI SDK 自动拼 `/chat/completions`）
- 协议：**OpenAI Compatible**（官方文档明示推荐用 OpenAI SDK + 改 baseURL）
- 模型（**新主流**）：
  - `deepseek-v4-pro`：思维链（思考模式），用于首次解读 / 复杂分析
  - `deepseek-v4-flash`：高速 + 便宜，用于多轮对话
- 模型（**即将弃用 2026/07/24**）：
  - `deepseek-chat` → 对应 `deepseek-v4-flash` 非思考模式
  - `deepseek-reasoner` → 对应 `deepseek-v4-pro` 思考模式
- SSE：`stream: true`；OpenAI SDK 已封装 `[DONE]` / keep-alive 处理，无需手写解析
- 超时：建议透传 `signal`，Drawer 关闭或路由切换时主动 abort
- CORS：官方支持 Web 端直发（截至 2026-04），policy 变更需关注 [api-docs.deepseek.com](https://api-docs.deepseek.com/)
- 关键参数（v4 系新增）：
  - `thinking: { type: 'enabled' }` —— 启用思考模式
  - `reasoning_effort: 'low' | 'medium' | 'high'` —— 思考深度
  - 设置页第二期可暴露这两个开关，MVP 先固定 `enabled + medium`
