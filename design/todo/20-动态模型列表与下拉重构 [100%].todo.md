## 0. 模块定位

todo 19 让设置页接入 8 家 Provider，但用户在体验中陆续反馈 3 个问题：

1. **下拉组件不对**：Phase 3 用过 reka-ui Select / Popover+Command 两种实现，体验都不如项目里早就用过的「选择城市」`CityCombobox`（无 portal、滚动稳、风格和站点统一）
2. **模型列表会过期**：registry 里硬编码的 model id 列表 1-3 个月就老化一次，新模型无法被尝试，老用户的 storage 残留旧 id 还会触发 sanitize 兜底
3. **温度旋钮太显眼**：占卜场景下温度默认 0.3 已偏高，绝大多数用户不需要调；首屏铺开的 Slider 让设置页主线变得拖沓

本 TODO 把这三件事一并处理，**不破坏 todo 19 已交付的 LlmProvider 契约 / 8 家配置 / BYOK 直发隐私模型**。

**非目标**：

- ❌ 不做"模型自动同步"——仍由用户主动点 [刷新模型]
- ❌ 不做远程模型缓存的 TTL 自动失效——sessionStorage 关闭浏览器后失效已是合理界限
- ❌ 不引入新 SDK——8 家全部用 todo 19 装的 SDK 现成接口（OpenAI `client.models.list` / Anthropic `client.models.list` / Google `client.models.list`）
- ❌ 不改隐私页文案——本期改动对 BYOK 直发模型零影响，隐私描述不需要重写

## 3. 关键决策与权衡

### 3.1 SearchableCombobox vs 继续用 reka-ui

**决策**：自建。

**理由**：
- reka-ui Select 不能搜索；Popover+Command 能搜索但 portal 让样式不在组件层级、scroll 抖动、与 CityCombobox 视觉双标
- 自建 200 行代码可控；交互完全对齐 CityCombobox，用户不需要学第二种下拉
- 体积代价小：feat-ai chunk 增加几百字节
- 退回 reka-ui 的成本随时低（contract 只暴露 `modelValue / options / labelOf / matchFn`）

**反方意见**（已否决）：
- "应该用业界标准库"：项目本来就有 CityCombobox 一例，新组件作为它的泛型化版本，不算另起炉灶
- "shadcn 风格更现代"：实际选型时用户明确要求"参考选择城市那种"

### 3.2 远程模型缓存：sessionStorage vs localStorage

**决策**：sessionStorage。

**理由**：
- 模型清单是"短保鲜"数据，厂商每 1-3 月会增/弃模型；localStorage 持久化容易让用户长期看到 stale 列表
- 多 tab 时 sessionStorage 隔离，避免一个 tab 的旧 cache 污染另一个 tab 的"刚刚刷新"
- 浏览器关闭后失效 = 自然提醒"重新评估"
- 隐私：API key 已在 localStorage（必须长期存）；模型清单不需要长期存

**反方意见**（已否决）：
- "永久缓存能减少 API 调用"：用户主动点刷新时才拉，不存在自动 fetch；"减少 API 调用"是伪需求

### 3.3 启发式 tag vs 厂商 metadata

**决策**：用启发式（关键字 → tag）。

**理由**：
- 8 家 SDK 的 model 元数据格式各不相同：OpenAI 几乎只给 id+created；Anthropic 给 display_name；Gemini 给 inputTokenLimit + supportedActions + thinking
- 把这些字段拼成统一的 `tags` 字段成本高且易错
- 厂商命名规律是观察值（flash=fast、haiku=fast+cheap、reasoner=thinking、coder=coding 等），过去 3-4 年稳定
- 即使将来命名变化，inferModelTags 退化为"返回空数组"即可，不影响功能

**反方意见**：
- "纯英文关键字会漏中文模型"：实测 8 家的 model id 全部用英文 / 拼音英文（kimi-k2.6 / qwen3-max / glm-4.6），无中文 id
- "假阳性怎么办"：tag 仅用于 hint 描述，不影响实际 chat 行为；偶尔多/少一个 tag 不构成 bug

### 3.4 Temperature 默认值 0.3 → 0.2

**决策**：降到 0.2。

**理由**：
- 占卜解读对"同一命盘多次解读应一致"的诉求 > "回答有创意"
- 0.2 仍然不是 0（确定性输出会显得僵硬），保留少量随机性
- 老用户 storage 中的自定义值（0.1 / 0.3 / 0.7 等）不被覆盖，仅新用户首次使用时拿到 0.2

### 3.5 Temperature 默认折叠

**决策**：自定义 button + v-show（不用 `<details>` 原生标签）。

**理由**：
- 原生 `<details>` 默认箭头与 shadcn 主题不统一，需要重写
- 自定义 button 可以同时显示当前值徽标，让收起状态也能一眼看到当前值，无需展开
- 状态用本地 ref（不持久化），下次进入回到默认收起，避免老用户的展开偏好覆盖新用户

## 4. 风险与回滚

### 4.1 主要风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 厂商 SDK 升级后 `models.list` 字段重命名 | 中 | listModels 拉空 / 报错 | provider 实现统一 try/catch → toLlmError；store 失败不污染缓存；UI 红字提示用户回退到内置列表 |
| 启发式 inferModelTags 误标 | 高 | tag hint 描述偏差 | tag 仅影响显示文案，不影响 chat 行为；用户可凭 tag 描述去厂商官网核对 |
| sessionStorage 在某些 webview 中被禁用 | 低 | useStorage fallback 到内存 | useStorage mergeDefaults 保证 storage 不可用时回退到默认 `{}`，UI 仍显示 "使用内置列表" |
| 老 storage 中残留 deprecated 模型 id | 已接受 | sanitize 兜底回 default | 与 todo 19 P1-03 同样的 sanitize 路径 |

### 4.2 回滚路径

每个 commit 独立可回滚：

- **Commit 1 回滚**：`git revert 0348c8c` 恢复 reka-ui Combobox（CityCombobox 早就在用，无副作用）
