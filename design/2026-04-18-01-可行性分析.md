## 1. 项目边界与核心约束

### 1.1 项目目标

做一个"算命占卜"主题的 **静态网页**，用户无需注册、无需后端，打开即用。

### 1.2 可行性评估的硬约束

所有方法都必须满足以下约束才"可行"：

| 约束 | 说明 | 违反则判定 |
|------|------|-----------|
| **纯前端实现** | 所有算法、数据都在浏览器内运行 | ❌ 不可行 |
| **离线可跑** | 不依赖任何外部 API / 后端服务 | ❌ 不可行 |
| **体积可控** | 单模块打包后 gzipped < 200 KB | ⚠️ 需优化 |
| **移动端友好** | 交互能在手机上顺畅完成 | ⚠️ 降档 |
| **文化娱乐定位** | 不涉及严肃预测、医疗、财务建议 | ⚠️ 合规风险 |

### 1.3 与方案设计的区别

- **本文档（可行性）**：判断"每种方法能不能做、做到什么程度"
- **后续方案设计**：基于可行性结论，选组合、画原型、定数据结构

## 2. 评分方法说明

每种方法会在 **5 个维度** 上打分（1-5 分，5 分最好）：

| 维度 | 关注点 | 5 分标准 |
|------|--------|---------|
| **技术可行性** | 纯前端能否实现？有无现成库？ | 算法简单 + 有成熟 npm 库 |
| **数据可行性** | 需要多少静态数据？体积合理吗？ | 数据量 < 50 KB |
| **算法准确度** | 前端实现能否做到"可接受的准确"？ | 和专业版差异可忽略 |
| **UI 展示可行性** | 能否在网页上直观呈现结果？ | 简单卡片/图表即可展示 |
| **开发成本** | 从零实现到可用的工作量 | < 3 天 |

**综合档位**（基于加权评分）：

- 🟢 **强推荐**（总分 ≥ 22）：技术、数据、展示都很友好，建议首批开发
- 🟡 **可做**（总分 16-21）：有一定门槛但可控，适合作为增强功能
- 🟠 **勉强**（总分 11-15）：技术或数据瓶颈明显，需要较大投入
- 🔴 **不建议**（总分 ≤ 10）：硬约束违反或成本严重失衡，建议放弃

## 3. 19 种方法逐一分析

### 3.1 入门级方法（4 种）

---

#### 方法 1：称骨算命（袁天罡称骨法）

> 将生辰的年、月、日、时各对应一个"重量"，相加得总骨重，查歌诀解读。

| 维度 | 分 | 说明 |
|------|---|------|
| 技术可行性 | 5 | 纯查表 + 加法，十几行代码即可 |
| 数据可行性 | 5 | 4 张重量表 + ~51 条歌诀，总计 < 10 KB |
| 算法准确度 | 5 | 无模糊空间，严格按原法则 |
| UI 展示可行性 | 5 | "一杆秤"+ 歌诀卡片即可，可做动画 |
| 开发成本 | 5 | 1-2 天 |

**总分：25 / 25** — 🟢 **强推荐**

**一句话结论**：教科书级别的静态网页友好方法，零阻力，必做。

---

#### 方法 2：小六壬（掐指一算）

> 根据农历月、日、时辰三个数字在六宫中推移，得到吉凶宫位。

| 维度 | 分 | 说明 |
|------|---|------|
| 技术可行性 | 5 | 算法 5 行代码，取模循环即可 |
| 数据可行性 | 5 | 6 宫 × 4 维度解读 ≈ 5 KB |
| 算法准确度 | 5 | 算法确定性高 |
| UI 展示可行性 | 5 | 6 宫环形图 + 命中宫位高亮，视觉效果佳 |
| 开发成本 | 5 | 1-2 天 |

**总分：25 / 25** — 🟢 **强推荐**

**一句话结论**：做"每日运势"入口的完美选择，用系统当前日期就能自动出结果。

---

#### 方法 3：抽签灵签（观音灵签等）

> 从签文库随机抽一签，显示四句诗和白话解读。

| 维度 | 分 | 说明 |
|------|---|------|
| 技术可行性 | 5 | `Math.random()` + 查表 |
| 数据可行性 | 4 | 观音灵签 100 签 ≈ 30-50 KB，可懒加载 |
| 算法准确度 | 5 | 随机本就是其设计 |
| UI 展示可行性 | 5 | 签筒摇晃 + 签条展示，仪式感强 |
| 开发成本 | 5 | 1-2 天（数据录入是主要工作量） |

**总分：24 / 25** — 🟢 **强推荐**

**一句话结论**：仪式感最强的一种，签文数据可网上找到公开版本，开发无难度。

---

#### 方法 4：诸葛神数

> 用户说出 3 个 1-9 的数字，按公式算出 1-384 的签号，查签诗。

| 维度 | 分 | 说明 |
|------|---|------|
| 技术可行性 | 5 | 公式明确，简单计算 |
| 数据可行性 | 3 | 384 签 × 每签文本 ≈ 80-120 KB |
| 算法准确度 | 5 | 公式唯一，无争议 |
| UI 展示可行性 | 4 | 3 个数字输入 + 签诗卡片，略单调 |
| 开发成本 | 3 | 2-3 天（384 条数据录入较费力） |

**总分：20 / 25** — 🟡 **可做**

**一句话结论**：体量比抽签大一个量级，数据录入是瓶颈，可作为抽签的"高级版"补充。

### 3.2 主流命理方法（8 种）

---

#### 方法 5：八字（四柱）命理【v1.3 MVP 恢复增强版】

> 将生辰转为天干地支八字，分析五行、十神关系。**MVP 增强版 = 命盘 + 五行 + 十神 + 大运（10 年一运） + 流年（当年干支）**，即 v1.1 完整版，专业度作为产品核心卖点。

**MVP 范围评分（增强版：含大运/流年）**：

| 维度 | 分 | 说明 |
|------|---|------|
| 技术可行性 | 5 | `tyme4ts` 一行代码出四柱、十神、五行、纳音，`.getYun(gender)` 出大运 |
| 数据可行性 | 4 | 解读模板：十神（10 段）+ 五行分析 + 大运吉/中/凶标签（~100 段）≈ 80 KB |
| 算法准确度 | 5 | 库的排盘和大运起运算法已在社区验证多年（作者 6tail 是这块权威） |
| UI 展示可行性 | 4 | 四柱表 + 五行雷达 + 大运时间轴，桌面体验完整，移动横向滑动 |
| 开发成本 | 4 | 增强版 8-10 天（排盘 1 天 + 解读文案 4-5 天 + 大运时间轴 2-3 天）|

**总分：22 / 25** — 🟢 **强推荐**（MVP 范围）

**技术可行性依据**：
- `tyme4ts` 官方 README 明确列出：**大运（Great luck）、小运（Small luck）、童限（Childhood years）、身宫（Bazi palace）** 均原生支持
- `lunar-typescript` 的 `EightChar` 类提供 `getYun(gender)` 方法直接返回大运起运信息
- 流年即当年的干支，通过 `LunarYear.fromYear(2026).getSixtyCycle()` 一行代码
- **算法零风险**，产品方决策"专业度优先于简化"

**MVP 做哪些大运/流年内容**：
- **大运时间轴**：展示 8-10 个十年大运，每个大运显示干支 + 起止年龄 + 吉/中/凶标签 + 一句话提示
- **流年标签**：当年（如 2026 丙午）相对日主的五行关系 + 一句话财运/事业/感情提示
- **不做**：流月、流日、分面（事业/感情/财富/健康）深度分解 — 这些作为后续扩展

**风险与对策**：
- 真太阳时（按出生地经度修正）需要额外经纬度数据，默认关闭作为高级选项
- 大运吉/中/凶标签依赖"大运干支 vs 日主喜用神"规则，建议 LLM 模板化 + 人工审校 100 段核心组合
- 子时换日（晚子 / 早子）在不同流派有分歧，`tyme4ts` 支持配置，需明确采用何种规则
- 大运起运时间（含"起运前"的童年段）有多种算法，`tyme4ts` 可自定义算法
- 流年解读同样易出现 LLM 幻觉，建议先做 60 年（60 甲子循环）× 5 旺衰档 = 300 段模板的基础覆盖，然后才开放流年功能

#### 方法 6：紫微斗数（v1.4 升档为 MVP 专业模块）

> 排出 12 宫 + 14 主星 + 六吉六煞 + 杂耀共 100+ 颗星，分析星曜在宫位的组合论命。含大限/小限/流年/四化。

| 维度 | 分 | 说明 |
|------|---|------|
| 技术可行性 | 5 | [`iztro`](https://github.com/SylarLong/iztro) v2.5.8（2026-03，3.5K ⭐，MIT，周下载 4K）原生输出 12 宫完整星盘 + 大限 + 流年 + 四化，完全纯前端 |
| 数据可行性 | 3 | 14 主星 × 12 宫 × 两性 ≈ 336 条命宫主星模板；六吉六煞 × 12 宫 ≈ 144 条；四化 × 10 干 = 40 条；合计 500+ 条解读，工作量可控 |
| 算法准确度 | 5 | iztro 是目前 JS 生态最成熟的紫微库，用户量大、流派兼容（可切换飞星派/三合派）、测试覆盖充分 |
| UI 展示可行性 | 3 | 桌面 4×4 宫盘布局成熟；移动端挑战大但可降级（"主要 4 宫+点击展开"的交互） |
| 开发成本 | 2 | 约 8-10 人日：UI 约 4 天（两套风格 × 桌面/移动盘）、数据约 3 天（主星 + 四化模板）、联调 1-2 天 |

**总分：18 / 25** — 🟡 **可做**（与八字并列为专业支柱）

**一句话结论**：有了 `iztro` 之后，技术门槛已消失；只要接受"移动端分宫展开而非全屏盘"的交互降级，紫微斗数完全可以作为 **MVP 专业版**的核心模块之一，与八字形成"双擎"覆盖命理受众。

**MVP 范围**：
- 12 宫完整排盘（4×4 桌面盘 / 移动分宫展开）
- 14 主星亮度展示
- 六吉六煞 + 重要杂耀（禄存、天马、天魁天钺、左辅右弼、文昌文曲、四煞）
- 本命四化（化禄/化权/化科/化忌）
- 命宫主星简析 + 身宫 + 命主身主 + 五行局
- 大限（10 年一限）+ 流年（当年干支 + 四化）

**排除范围**（Post-MVP）：
- 飞星/三合等高级流派
- 小限（与流年重叠度高）
- 流月/流日/流时
- 宫位两两对冲的"三方四正"复杂组合论
- 合盘

## 4. 知识库来源分析（v1.1 新增）

> **核心问题**：这些占卜方法的"算法"和"数据"到底从哪里来？模型能编吗？有没有库？要自己录入吗？
>
> **核心结论**：**分三类来源组合获取**。算法类用成熟 npm 库（零风险）；大量静态文本用公开数据采集（GitHub 爬虫项目/开放 API 一次性导出）；解读文案用 LLM 生成 + 人工审校。

### 4.1 知识库来源的三类

| 类型 | 含义 | 示例 | 信任度 |
|------|------|------|--------|
| **A 类：算法库** | 成熟 npm 包，算法+基础数据打包好 | `tyme4ts`、`iztro`、`najia-core` | ⭐⭐⭐⭐⭐ 可直接用于生产 |
| **B 类：公开数据** | 签文原文、歌诀、数理表等**历史公共文献** | 观音灵签 100 签签诗、袁天罡歌诀、81 数理 | ⭐⭐⭐⭐ 需清洗校对 |
| **C 类：解读文案** | 非原文的白话解释、建议、引导语 | "十神含义""大运怎么看"等 | ⭐⭐⭐ LLM 生成 + 人工审校 |

### 4.2 A 类：算法库（可直接 npm 安装）

经过调研，以下库**当前活跃维护**（2025-2026 年仍在更新）：

| 库 | 覆盖功能 | 周下载 | 许可 | 核心 API 示例 |
|----|---------|-------|------|--------------|
| **`tyme4ts`**（强推荐） | 公历/农历/干支/节气/八字/**大运/小运/童限**/身宫 | ~2.5K | MIT | `SolarTime.fromYmdHms(...)` → `.getLunarHour().getEightChar()` → `.getSixtyCycle()` |
| **`lunar-typescript`**（6tail 经典） | 同上 + 吉神/喜神/福神/财神 | ~18K | MIT | `Lunar.fromDate(date)` → `.getEightChar()` → `.getYun(gender)` |
| **`iztro`** | 紫微斗数（12 宫 + 运限：大限、小限、流年、流月、流日、流时） | ~4.1K | MIT | `astro.bySolar('2000-8-16', 2, '女')` |
| **`najia-core`** | 六爻京房纳甲（起卦 + 断卦分析） | 较少 | MIT | `getNaJiaData()` / `generateRandomHexagram()` |
| **`chinese-character-strokes`** | 汉字笔画（约 20000 字）| 较少 | MIT | 查询指定字的笔画数 |

**结论**：八字、紫微、六爻、姓名学的**算法**层面，**无需自研**，直接 npm install。

### 4.3 B 类：公开文献数据（需一次性采集）

这部分是**本项目真正的数据工程**，涉及的方法有：

| 方法 | 所需数据 | 规模 | 可得性 |
|------|---------|------|-------|
| 观音灵签 | 100 签签诗 + 解曰 + 仙机 + 典故 | ~100 KB | ⚠️ **没有现成 GitHub JSON 库**；有免费 API（可一次性爬取）；有公开网站（爬虫项目可参考） |
| 诸葛神数 | 384 签签文 | ~80 KB | ⚠️ 同上，已有开源 Python 爬虫项目（`zhugehorary.streamlit.app`）作参考 |
| 称骨算命 | 年/月/日/时重量表 + 51 首歌诀 | ~10 KB | ⚠️ 文献广泛流传，需手工整理为 JSON |
| 姓名学 81 数理 | 1-81 每个数字的吉凶描述 | ~30 KB | ⚠️ 文献公开，需手工整理 |
| 周公解梦 | 约 2000-9000 条词条 | 100-500 KB | ✅ **已有开源数据集**（CSDN 博客分享 9000 条版本；Hugging Face 有 fortune-telling 数据集） |
| 六爻爻辞 | 64 卦 × 6 爻 = 384 爻辞 + 卦辞 | ~80 KB | ✅ **公版文献**（《易经》）；`chinese-poetry` 等项目有整理版 |

**获取策略**：
1. **优先复用**：先搜 GitHub/Gitee/Hugging Face 是否有现成 JSON 数据集
2. **次选采集**：若无，写一次性爬虫从公开网站（如 `api.t1qq.com` 免费 API、`chazidian.com` 等）拉取并存为静态 JSON，打包进项目
3. **人工清洗**：采集后需人工抽样校验（文字错漏、标点、内容一致性）
4. **合规处理**：采集的文本都是**公版文献**（古籍、歌诀、传统数理），无版权风险；但需声明数据来源

**数据工程量估算**（如选综合门户版，含大运/流年）：
- 观音灵签：爬取 + 清洗 ≈ 1 天
- 称骨 51 首歌诀：手工录入 + 校对 ≈ 1 天
- 81 数理：手工录入 ≈ 0.5 天
- 周公解梦：现成数据集导入 ≈ 0.5 天
- **大运吉/中/凶**：LLM 生成 100 段 + 人工审校 ≈ 2 天
- **流年模板**：LLM 生成 300 段（60 甲子 × 5 旺衰档）+ 人工审校 ≈ 3 天
- **合计：约 8 天** 一次性工作

### 4.4 C 类：解读文案（LLM 生成 + 人工审校）

这部分是"原文之外的白话解释、用户引导、现代化解读"。举例：

| 场景 | 模型可否直接生成 | 建议做法 |
|------|----------------|---------|
| 八字十神含义（如"正官是什么意思"） | ✅ 可以，基础知识通用 | LLM 批量生成初稿 → 人工校对术语准确性 |
| 大运/流年的查询指引（如"2026 年您走什么运"）| ✅ 可以 | LLM 生成模板，用户数据动态填充 |
| 五行平衡建议（如"水弱需要多接触蓝色"） | ⚠️ 有编造风险 | LLM 生成 + 保守措辞（"可尝试"而非"必须"）|
| 小六壬 6 宫 × 4 维度解读（共 24 段） | ✅ 通用易生成 | LLM 生成 → 审校 → 固化为静态数据 |
| 签文白话翻译（签诗是古文）| ⚠️ 可能失真 | LLM 生成 + 古文专业人士抽检 |
| 签文现代化建议（如"这签抽到适合做什么"）| ⚠️ 易过度发挥 | LLM 生成 + 保守措辞 + 加免责声明 |

**风险点**：
- LLM 可能"编造"不存在的术语或解读（幻觉）
- 命理文化敏感词需要把关（避免迷信宣传违规）
- **原则**：**原文/古籍**必须用公开文献，**解读**可以用 LLM 但**必须注明"现代化解读仅供参考"**

### 4.5 本项目建议的数据策略

```text
┌─────────────────────────────────────────────────────────────┐
│                      数据层分层建议                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【算法层】npm 库直接用                                      │
│    → tyme4ts （八字、大运、流年核心）                        │
│    → iztro   （紫微斗数，如做）                              │
│    → najia-core（六爻，如做）                                │
│                                                             │
│  【原文层】一次性采集 → 静态 JSON，打包入项目                │
│    → src/data/lingqian/guanyin.json   （观音灵签 100 签）    │
│    → src/data/chenggu/poems.json      （称骨 51 歌诀）       │
│    → src/data/xingming/numerology.json（81 数理）            │
│    → src/data/jiemeng/zhougong.json   （周公解梦）           │
│    （所有文本均为公版古籍文献，无版权风险）                  │
│                                                             │
│  【解读层】LLM 生成 → 人工审校 → 固化为静态数据              │
│    → src/data/bazi/tenGods.ts         （十神解读 10 段）     │
│    → src/data/bazi/fortuneHints.ts    （大运吉/中/凶 100 段）│
│    → src/data/bazi/flowYearHints.ts   （流年 60 甲子×5档=300 段）│
│    → src/data/liuren/palaces.ts       （小六壬 24 段）       │
│    → 所有"白话解读"类文本                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心结论**：
- ✅ **算法零风险**：所有可做的方法都有成熟开源库
- ✅ **原文有保障**：公版文献，无版权问题，3 天可全部录入
- ⚠️ **解读需把关**：LLM 生成后必须有人审核，避免术语错误和过度发挥
- ✅ **总体可行**：组合 A/B/C 三类来源，知识库建设不构成项目瓶颈

**包含**：组合 A 的 4 项 + **八字命理（含大运 + 流年）** + **紫微斗数（12 宫 + 四化 + 大限）** + 姓名学 + 解梦

- 开发周期：**约 8-9 周**（+2 周紫微：UI 4 天 + 数据 3 天 + 联调 2 天）
- 体积：< 700 KB（按模块懒加载，紫微本身 ~80 KB + 解读模板 ~150 KB）
- 适合：中期上线、覆盖广泛用户，"双引擎命理"是对标主流算命站的差异化卖点
- **推荐理由**：
  - 覆盖"1 分钟玩"到"15 分钟深度测"的完整体验梯度
  - **八字 + 紫微命理双擎**：八字擅长"先天格局 + 十年一运"，紫微擅长"12 人生宫位 + 本命四化"，互补覆盖命理受众
  - **算法零风险**：`tyme4ts` + `iztro` 两库合计 10K+ ⭐、MIT 许可、测试完善
  - 8 个模块全部在 🟢🟡 档，技术风险可控
  - 可分阶段交付（先 4 个入门→加八字→加姓名+解梦→加紫微）
  - 知识库来源清晰，模板数据工程是主要成本（约 900-1000 段模板 + 审校）
- **关键降级策略**：
  - 紫微 4×4 宫盘在移动端采用"主要 4 宫（命/财/官/夫）+ 点击展开其余 8 宫"交互
  - 紫微主星模板用"同类辅星归并"策略：14 主星 × 12 宫覆盖主干；辅星用"加点套话"模块化
- 知识库来源：
  - A 类：`tyme4ts`（八字+大运+流年+择日）、`chinese-character-strokes`（姓名笔画）、`iztro`（紫微 12 宫 + 大限 + 四化 + 流年）
  - B 类：观音灵签 100 签、称骨 51 首歌诀、81 数理、周公解梦词条
  - C 类：八字十神 100 段、大运 100 段、流年 300 段、小六壬 24 段、**紫微 14 主星 × 12 宫 336 段、四化 × 10 干 40 段、六吉六煞 × 12 宫 144 段**（LLM 生成 + 审校）

#### 组合 C：极客深度版（专业向）

**包含**：组合 B + **八字分面深度解读**（事业/感情/财富/健康） + **紫微飞星三合论** + 梅花易数起卦 + 六爻起卦

- 核心卖点：传统术数大全，超专业感

- 开发周期：约 12-14 周

  - 受众较窄

### 5.4 我的推荐（v1.4 更新）

**强烈推荐选择【组合 B 综合门户 + 命理双擎版】作为下一步方案设计的起点**，理由：

1. **命理双擎 = 差异化核心卖点** — 市面主流算命站多只专精一种（八字或紫微），tt-qimen 双擎互补覆盖命理受众
2. **八字 MVP 含大运 + 流年** — `tyme4ts` 原生支持，算法零风险
3. **紫微 MVP 含 12 宫 + 四化 + 大限** — `iztro` 原生支持，算法零风险
4. **风险最小** — 8 个模块全部在 🟢🟡 档，无技术障碍
5. **可分阶段** — 先做组合 A（4 模块）上线，再迭代加八字→姓名+解梦→紫微斗数
6. **体验完整** — 覆盖"1 分钟娱乐"到"15 分钟深度测"全梯度
7. **知识库可控** — 三类来源（算法库 / 公版原文 / LLM 解读）都有成熟应对方案
8. **体积可控** — 配合懒加载，首屏可保持在 180 KB 以内（紫微按需加载）
9. **扩展路径清晰** — 后续可无损升级到组合 C（加入八字分面深度 + 紫微飞星 + 其他专业术数）

## 6. 风险与注意事项

### 6.1 技术风险

| 风险项 | 影响 | 缓解措施 |
|-------|------|---------|
| 农历转换边界（闰月、19XX 前数据） | 八字、称骨、小六壬全受影响 | 用 `tyme4ts` / `lunar-typescript`（支持 1900-2100）|
| 子时换日规则分歧（早子 / 晚子） | 八字排盘不同流派结果不同 | `tyme4ts` 支持配置，文档明确采用规则 |
| 真太阳时 | 八字进阶用户可能要求 | 默认关闭，作为高级选项 |
| 大运起运算法多流派 | 起运时间小幅差异 | `tyme4ts` 可自定义算法；默认用通用算法 |
| 康熙字典笔画歧义 | 姓名学核心 | 使用 `chinese-character-strokes`（覆盖 20000 字）|
| 数据体积超标 | 首屏加载慢 | 按模块懒加载 + JSON 压缩（gzip + brotli） |

### 6.2 知识库风险（v1.1 新增）

| 风险项 | 说明 | 应对 |
|-------|------|------|
| **LLM 解读幻觉** | 生成的十神/五行解读可能混淆术语 | 所有 LLM 产出必须人工审校；保守措辞 |
| **签文版本差异** | 观音灵签民间流传有多个版本 | 明确标注采用的版本来源；避免混用 |
| **采集数据质量** | 爬虫结果可能有错字、标点错误 | 采集后人工抽样校对 10% 样本 |
| **公开数据许可** | 部分免费 API 有调用限制 | **只用于一次性导出**，不在线调用（静态约束） |
| **康熙笔画版本** | 不同来源笔画数可能冲突 | 采用 `chinese-character-strokes`（基于 UnicodeCJK-WuBi），文档中声明数据源 |

### 6.3 内容与合规风险

| 风险项 | 说明 | 应对 |
|-------|------|------|
| 算命敏感性 | 不同地区对"算命"内容审核不同 | 显著位置加"仅供娱乐、文化体验"声明 |
| 不准确的预测误导 | 用户可能真信 | 避免"必定""一定"等绝对化表述 |
| 医疗/财务建议 | 签文可能涉及健康、投资 | 对这类解读添加"请咨询专业人士"提示 |
| 隐私 | 生辰信息敏感 | 所有数据仅本地存储（localStorage），不上传 |
| 版权 | 引用签文/歌诀 | 均为古籍公版文献，无版权风险；但需声明来源 |

### 6.4 视觉与体验风险

| 风险项 | 应对 |
|-------|------|
| 国风视觉过度 → 年轻用户排斥 | 采用"现代简约 + 中式点缀"而非全套古风 |
| 移动端排盘显示不下 | 八字简化为 4 列 + 关键信息，详细信息折叠 |
| 大运时间轴太长 | 大运通常 8-10 个，采用蛇形网格（桌面 5×2 / 移动 2×5）+ 虚线连接，避免横向滚动 |
| 紫微 12 宫移动端挤压 | 主要 4 宫（命/财/官/夫）默认全展示，其余 8 宫折叠为列表，点击展开 |
| 动画性能差 | CSS 动画为主，避免复杂 JS 动画 |

## 7. 结论

### 一句话总结

**19 种占卜方法中，10 种在静态网页下可行（🟢🟡档）。推荐采用【组合 B 综合门户 + 命理双擎版】作为 `tt-qimen` 项目的目标方案 —— 称骨、小六壬、观音灵签、八字（含大运 + 流年）、紫微斗数（12 宫 + 四化 + 大限）、择日、姓名学、解梦共 8 个模块。核心技术栈：Vue 3 + Vite + `tyme4ts` + `iztro` + `chinese-character-strokes` + 静态 JSON 数据层。**
