## 0 · 改造原则（先定调，再动手）

> 需求原话："我希望所有的注释没有问题，现代化翻译的是必须的，但是需要有出处的原文体现。"

落地为四条强约束：

| # | 原则 | 落地方式 |
|---|---|---|
| P-1 | 所有用户可见文案 = 现代化译文 + 古籍原文出处（两者并列） | 数据层双字段：`interpret`（现代译文）+ `classical`（原文片段）+ `source`（书名 · 卷章 · 行号锚点） |
| P-2 | 没有古籍原文的字段必须显式标注"非古籍" | 数据层字段：`isClassical: false` + `source: '编者撰写'`；UI 层不显示出处行，但保留数据层痕迹便于审计 |
| P-3 | 沿用已有的 `PATTERN_LONG_INTERPRET` / `SHENSHA_MEANING` 的双层结构模式，统一全项目 schema | 见 § 1 目标 schema |
| P-4 | 现代化翻译必须过两道校：① 与原文语义对齐 ② 不夹带现代职业/行业硬配对 | 每个字段的 `interpret` 由 raw 原文直译，而非二次扩写 |
| P-5 | `classical` 字段**字迹简繁 = raw 原文字迹**，**不做简繁转换**；`source` 只写"《书名》·卷/篇·L起行-止行"，不混入代码侧差异注记 | 按 raw 实际字迹落字；code/raw 名称差异走**代码注释**，不向用户展示（用户决策 2026-04-22） |

**P-5 派生规则**（按 raw 字迹定，不按团队偏好定）：

- `nayin.ts` → raw = 《三命通会 · 卷一》维基文库版（**简体**）→ classical **简体**

- `tenGods.ts` → raw = 《三命通会 · 卷五 / 卷六》维基文库版（**简体**）→ classical **简体**

## 1 · 目标 schema（全项目统一）

所有面向用户的文案字段都应能映射到如下形状（视字段密度分 Full / Lite 两种）：

### 1.1 Full schema（长解读字段）

```ts
interface SourcedContent {
  /** 现代化译文 — 用户可见主体 */
  interpret: string
  /** 古籍原文片段（关键句子，≤ 40 字） */
  classical?: string
  /** 出处锚点："《xxx》· 卷x · Lxxx-yyy" */
  source?: string
  /** 是否古籍可溯源。false 时 UI 不显示 classical 行，数据层留痕 */
  isClassical: boolean
  /** 审校状态 */
  auditStatus: 'verified' | 'pending' | 'non-classical'
}
```

### 1.2 Lite schema（短释义字段）

```ts
interface ShortSourcedContent {
  text: string
  source?: string
  isClassical: boolean
}
```

**现存已满足 schema 的文件**：

| 文件 | schema 合规度 | 备注 |
|---|---|---|
| `patternLongInterpret.ts` | 🟡 部分 | 有 `PATTERN_LONG_INTERPRET_STATUS` + 顶部注释锚点，但每条文本无 `source` 字段；需拆分 |
| `shenshaMeaning.ts` | 🟢 完全 | 已有 `auditStatus` + `source` + `pending` 占位分组 |

**未满足 schema 的文件**：`tenGods.ts`、`interpretTemplate.ts`、`nayin.ts`、`mockBazi.ts`（见 § 2）

**P0 已补的 FILE 级元数据常量**（便于 UI/CI 机器化读取）：

- `tenGods.ts` → `export const TEN_GODS_IS_CLASSICAL = false`
- `interpretTemplate.ts` → `export const INTERPRET_TEMPLATE_IS_CLASSICAL = false`
