# 紫微文本内容校验报告

日期：2026-04-28

## 检查范围

- `src/locales/modules/ziwei.zh-CN.ts`
- `src/locales/modules/ziwei.zh-TW.ts`
- `src/locales/modules/ziwei.en.ts`
- `src/modules/ziwei/data/*.ts`
- `src/modules/ziwei/components/*.vue`
- `src/modules/ziwei/core/ziwei.ts`

## 校验结论

当前紫微模块的中文基础文案整体可读，三语 locale 主键结构未发现明显缺失；新增的 B/C 类数据也能被 TypeScript 类型体系覆盖到。

但存在 4 类需要处理的问题：固定示例断语会误导用户、英文资源仍有大量占位稿、英文界面会混入中文“四化”字样、zh-TW / en 仍明确标注为骨架或待审稿状态。

## 问题清单

### P0：命盘解读正文是固定示例，可能与真实命盘不一致

位置：

- `src/locales/modules/ziwei.zh-CN.ts`
- `src/locales/modules/ziwei.zh-TW.ts`
- `src/locales/modules/ziwei.en.ts`
- `src/modules/ziwei/components/InterpretCards.vue`
- `src/modules/ziwei/core/ziwei.ts`

现象：

- `InterpretCards.vue` 的正文固定读取 `ziwei.interpret.cards.*.text`。
- locale 中的正文是具体盘例，例如“紫微贪狼同坐于辰宫”“贪狼化禄坐命宫”“破军坐申”等。
- `buildInterpretCards()` 只动态生成 tags，正文不会随实际命盘变化。

影响：

- 任意用户排出的命盘都可能看到同一组具体断语。
- 正文内容可能与动态标签、实际宫位、四化落点不一致，属于高风险内容准确性问题。

建议：

- 短期：隐藏 `sectionInterpret` 或把正文改成不绑定具体星曜/宫位的通用提示。
- 中期：将正文改为基于 `chart.interpretCards` 派生的模板化文本。
- 长期：接入已经拆分的 `SoulPalace / PalaceMajor / MinorStars / SihuaReading / DecadalReview / YearlyReview` 数据，避免固定示例断语。

### P1：英文主星/副星入宫大数据仍是占位稿

位置：

- `src/modules/ziwei/data/palaceMajor.en.ts`
- `src/modules/ziwei/data/minorStars.en.ts`

现象：

- `palaceMajor.en.ts` 使用 `[EN draft] ${star} in ${palace} palace — interpretation pending professional translation.`
- `minorStars.en.ts` 使用同类 `[EN draft] ... pending professional translation.`
- 这两份数据分别覆盖 168 段主星入宫、144 段副星入宫，英文用户会直接看到占位稿。

影响：

- 英文界面的核心解读内容不可作为正式文案发布。
- 占位文本会破坏用户信任，也会让“英文已支持”的产品语义不成立。

建议：

- 若近期发布英文界面，至少先隐藏这两类英文长文解读，或 fallback 到中文并提示未翻译。
- 若保持展示，需要先补齐人工英文译稿，再去除 `[EN draft]` 与 skeleton 标记。

### P1：英文界面四化标签混入中文“化”

位置：

- `src/modules/ziwei/components/DecadalDetail.vue`
- `src/modules/ziwei/components/ZiweiYear.vue`

现象：

- 模板硬编码：`化{{ shortSihua(m.sihua) }}`
- 英文 locale 的 `sihuaShort` 是 `Lu / Quan / Ke / Ji`
- 英文界面最终会显示为 `化Lu`、`化Quan` 等混排文本。

影响：

- 英文界面中四化标签不自然，也和 `Hua-Lu / Hua-Quan` 的英文术语不一致。

建议：

- 新增 i18n formatter，例如 `ziwei.sihuaMarkFmt`：
  - zh-CN：`化{label}`
  - zh-TW：`化{label}`
  - en：`Hua-{label}`
- 组件统一通过 formatter 渲染，避免模板内硬编码中文。

### P2：英文/繁体数据仍明确标注为骨架或待专业审稿

位置：

- `src/modules/ziwei/data/index.ts`
- `src/modules/ziwei/data/soulPalace.en.ts`
- `src/modules/ziwei/data/sihuaReading.en.ts`
- `src/modules/ziwei/data/soulMaster.en.ts`
- `src/modules/ziwei/data/decadalReview.en.ts`
- `src/modules/ziwei/data/yearlyReview.en.ts`
- `src/modules/ziwei/data/*.zh-TW.ts`

现象：

- 多处注释明确写着 `SKELETON`、`placeholder`、`骨架`、`需专业校对`。
- `sihuaReading.en.ts`、`soulMaster.en.ts`、`decadalReview.en.ts`、`yearlyReview.en.ts` 虽非 `[EN draft]`，但仍是短摘要骨架，不是完整翻译。
- zh-TW 多数由简繁转换生成，尚未完成台港命理语境审校。

影响：

- 内容质量状态和产品表现存在差距；如果 UI 对用户展示为完整三语支持，存在预期不一致。

建议：

- 在发布标准中区分“结构已覆盖”和“文案已审稿”。
- 对 zh-TW / en 增加内容状态 checklist，审完一类再移除 skeleton 标记。

### P2：英文界面仍会展示中文星名、宫位干支或 fallback 标签

位置：

- `src/modules/ziwei/core/ziwei.ts`
- `src/modules/ziwei/components/InterpretCards.vue`
- `src/modules/ziwei/components/SihuaReadingView.vue`
- `src/modules/ziwei/components/SoulMasterView.vue`

现象：

- `buildInterpretCards()` 直接使用 iztro 输出的中文星名，并拼接 `借`、`空宫`、`清宫`。
- `SihuaReadingView` 的英文 subtitle 会把中文天干 `{stem}` 放进 `'{stem}-Stem Four Transformations'`。
- `SoulMasterView` 标题可以翻译，但星名展示仍来自 `chart.meta.mingZhu / shenZhu` 的中文名。

影响：

- 英文界面的标题和说明已翻译，但核心星名/标签仍中文混排。
- 对熟悉拼音术语的英文用户不够友好。

建议：

- 建立 star / stem / palace display name 映射，按 locale 输出：
  - zh-CN：贪狼、甲、命宫
  - zh-TW：貪狼、甲、命宮
  - en：Tanlang、Jia、Life Palace
- 避免在 core 层直接拼接用户可见中文词，优先输出结构化 key，交给 UI/i18n 渲染。

## 已验证项

- 三个 `ziwei.*.ts` locale 文件的主要占位符集合一致，未发现 `{name}` / `{age}` / `{count}` 这类 formatter 在某语言缺失。
- `sihuaReading` 的 10 天干 × 4 化结构完整，星名与 `STEM_SIHUA_TABLE` 口诀顺序一致。
- `soulMaster` 当前实现覆盖 13 颗候选星，和 `ZH_MING_SHEN_TO_KEY` 的 key 集合一致。
- `npm run type-check` 已执行；未出现紫微模块相关类型错误。

## 建议修复顺序

1. 先处理 P0：隐藏或重写固定示例断语，避免具体命盘内容误导用户。
2. 再处理 P1：英文占位稿与四化标签混排，决定英文界面是暂时隐藏、fallback 中文，还是补齐正式翻译。
3. 最后处理 P2：补 zh-TW 审校、英文星名/天干展示映射和内容状态标记。
