## 0. 模块定位

日本五格剖象法（1918 熊崎健翁创立）— 将姓名按笔画拆为天/人/地/外/总五格。
- 可行性 21/25 分 🟡 可做。
- 体验定位：**实用工具型** — 用户测自己名字 / 起名参考。
- 2026-04-23：**主体完工** — `src/modules/xingming/` 下 types/core/data/stores/components/XingmingPage 全量交付；81 数理、5 格算法、笔画修正表、复姓白名单、双主题、三语 i18n 均已上线。

### K.1 A — `lastComputed` 快照缓存

#### 设计

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 缓存对象 | **只缓存 input 快照**，不缓存 result 主体 | result 内含 i18n 解读文本（locale 切换会重译），缓存会陈旧；input → result 是纯函数，重算成本约 30ms（远低于 1.5s 骨架） |
| 比较口径 | `inputKey()` 字符串化 surname/givenName/gender/birthYear，`birthYear ?? 'null'` 兜底 | 避免 `undefined` vs `null` 误判为不同输入 |
| 失败路径 | `runCalculate` 失败时 `clearComputed()` | 避免下次刷新自动重现错误页 |
| 触发位置 | `XingmingPage.onMounted` 检查 `shouldRestore()` → `runCalculate(true)` + `revealImmediately()` | 跳过 1.5s 骨架直接展示结果；与 lingqian P4 lastResult 思路同源 |
| Storage key | `tt-qimen:xingming:lastComputed` | 与 input key 同 namespace，便于排查 |

#### 实现

| 类型 | 文件 | 内容 |
|------|------|------|
| 修改 | `stores/xingmingStore.ts` | 新增 `lastComputed: useStorage<XingmingInput\|null>` + `recordComputed` / `clearComputed` / `shouldRestore` + `inputKey` helper |
| 修改 | `XingmingPage.vue` | `onMounted` 走 `shouldRestore() → runCalculate(true) + skeleton.revealImmediately()`；`runCalculate` 成功路径调 `recordComputed`，失败/`onRecalculate` 路径调 `clearComputed` |

#### **隐性 BUG 发现 + 修复**

`useStorage<T \| null>` 当默认值传 `null` 时，vueuse **按 `null` 推断 serializer = boolean**（`StorageSerializers.boolean`），后续写 object 会走 `String(value)` 序列化成 `'[object Object]'`，刷新后恢复时 `JSON.parse` 失败，cache 形同虚设。

修复：显式传 `{ serializer: StorageSerializers.object }`，强制 JSON.stringify/parse 走 object 通道，同时 null 透传。

```ts
const lastComputed = useStorage<XingmingInput | null>(
  COMPUTED_KEY,
  null,
  undefined,
  { serializer: StorageSerializers.object },  // ⚠️ 关键：默认值 null 时必须显式
)
```

> 此 BUG **会影响项目里所有"默认值 null + 后续写 object"的 useStorage 调用**。建议日后类似场景一律显式 serializer，避免依赖 vueuse 的类型推断。

| P4 | 改 input.surname 但 lastComputed 不动 | `shouldRestore() === false`，不自动恢复；lastComputed 保留旧值（cache 未被消费） |
| P5 | onRecalculate | `lastComputed === null`；再刷新无结果区 |

### K.3 C — 多音字修正 UI · ADR：**不实现**

#### 决策

**取消该残留项，改在附录 E 中标注 `[—] 不实现`。**

#### 依据（用户跳过选项前，自主调研结论）

1. **算法层无需要**：康熙笔画体系按**字形**计算，**不依赖读音**。多数多音字（重 chóng/zhòng = 9 / 9，长 cháng/zhǎng = 4 / 4，和 hé/hè = 8 / 8）按任何读音笔画都相同。读音 → 笔画的映射在康熙体系下基本是 1:1。

2. **真正影响笔画的边缘 case 已被覆盖**：
   - **简-康熙差异**（如 张 = 張 = 11 画 / "于" → 康熙 "於" = 8 画）：`getKangxiCorrection` 修正表 ~200 字白名单已硬覆盖
   - **生僻字未收录**：`StrokesNotFoundError` + `xingming.computeError.rareChar` 友好态已上线
   - **简/康熙两套笔画并列展示**：`StrokesBreakdown.vue` 已经显示 "简 N · 康熙 N"，让用户自检差异

3. **维护成本不对称**：
   - 实现需维护一份"多音字 → {读音 → 字形 → 笔画}"映射表，业内**无公版数据源**
   - UI 需要 popover 切换 + 重算 + state 持久化 + i18n
   - 估算工时 ≥ 1.5 天；用户实际触发率（按真实多音字白名单 ~50 字 / 全体姓名用字 ~3000 字 ≈ 1.6%）远低于其他改进项

4. **业界共识**：康熙姓名学（熊崎流派）官方教材均按字形定笔画，未要求按读音切换；本模块 1-15 维度功能已超越 80% 在线"姓名测算"工具的覆盖度。

### K.4 vueuse useStorage object serializer 通用陷阱备忘

> 此处以 inline 文档形式提示其他模块开发同学

| 场景 | 危险 | 解药 |
|------|------|------|
| `useStorage<T \| null>(key, null)` | 推断 boolean serializer，写 object 变 `'[object Object]'` | `{ serializer: StorageSerializers.object }` |
| `useStorage<number \| null>(key, null)` | 同上 | `{ serializer: StorageSerializers.number }`（或 object） |
| `useStorage<T>(key, defaultObj)` | 默认值是 object 时 OK | 不需要显式 |

**项目内排查建议**：grep `useStorage<.*null` 和 `useStorage<.*\| null>` 模式，逐个确认 serializer 是否显式声明。
