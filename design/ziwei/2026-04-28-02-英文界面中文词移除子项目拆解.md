## 0. 问题陈述

当前 `core/ziwei.ts` 输出的 `ZiweiChart` 仍然在多个字段携带中文 string，导致 en locale 下 UI 渲染中英混排：

| 字段 | 中文示例 | en locale 当前显示 | 期望显示 |
|------|---------|-------------------|---------|
| `Star.name` | "贪狼"、"紫微"、"巨门" | "贪狼" | "Tan-Lang" / "Greedy Wolf" |
| `Brightness` | "庙"/"旺"/"陷" | "庙" / "陷" | "Temple-Brilliant" / "Trapped" |
| `InterpretCard.tags[].label`（chip） | "贪狼旺"、"借紫微"、"空宫"、"清宫" | "贪狼旺" | "Tan-Lang Vigorous" / "Borrowed Zi-Wei" / "Empty Palace" / "Clean Palace" |
| `Palace.ganzhi` | "庚辰" | "庚辰" | "Geng-Chen" |
| `Palace.changsheng12` | "长生"/"沐浴"/"养" | "长生" | "Chang-Sheng" / "Long-Life" |
| `DaxianMutagenStar.name` | "贪狼" | "贪狼" | "Tan-Lang" |
| `ZiweiMetaInfo.mingZhu / shenZhu` | "贪狼"/"巨门" | "贪狼" | "Tan-Lang" / "Ju-Men" |
| `ZiweiChart.fiveElementsClass` | "水二局"/"火六局" | "水二局" | "Water Stage 2 (Five Elements School)" |

> 这些字段大部分是 `string` 类型，未结构化为 `key`，所以 i18n 翻译路径不通——任何 component 拿到字符串后只能直接渲染。

## 1. 影响面（必须改的代码点）

### 1.1 类型层（必改）

`src/modules/ziwei/types/index.ts` 涉及类型扩展：

```ts
// 改前：
interface Star { name: string; ... }

// 改后：
import type { MajorStarKey } from '../data/soulPalace'
import type { MinorStarKey } from '../data/minorStars'
type StarKey = MajorStarKey | MinorStarKey | 'unknown'  // 'unknown' 兜底未识别副星

interface Star {
  /** 稳定 key（用于 i18n / 数据查表） */
  starKey: StarKey
  /** @deprecated 渐进迁移期保留，便于向后兼容；新代码用 starKey + i18n */
  name: string
  ...
}
```

### 1.2 数据查表层（必改）

`src/modules/ziwei/data/index.ts` 新增 `STAR_KEY_TO_I18N` 映射：

```ts
// 14 主星已有 ZH_STAR_TO_KEY（zh-CN → key），需反向：
export const STAR_KEY_TO_DISPLAY = {
  'ziwei':    { 'zh-CN': '紫微',    'zh-TW': '紫微',    'en': 'Zi-Wei' },
  'tianji':   { 'zh-CN': '天机',    'zh-TW': '天機',    'en': 'Tian-Ji' },
  // ... 14 主星
} as const
// 12 副星 + 6 煞星已有 ZH_MINOR_STAR_TO_KEY，同样需反向
```

或更优雅地走 i18n key：在 `ziwei.{lang}.ts` 加 `ziwei.starNames.{starKey}` 命名空间，UI 直接 `t(\`ziwei.starNames.${starKey}\`)`。

### 1.3 core 派生器（必改）

`src/modules/ziwei/core/ziwei.ts` 5 个派生器输出 `name: string` 改为 `starKey: StarKey`：

- `buildPalaces` → 每颗 Star 的 `name` 改为 `starKey`
- `buildSihuaMap` → `{ star: string }` 改为 `{ starKey: StarKey }`
- `buildInterpretCards` → 4 张卡的 `tags[].label` 改为结构化（`{ starKey, brightnessKey, sihuaKey, borrowed }`），UI 渲染时再 i18n 拼接
- `buildCurrentDecadal` → `mainStars[].name` 改为 `starKey`
- `resolveMutagenLandings` → `DaxianMutagenStar.name` 改为 `starKey`
- `buildMeta` → `mingZhu / shenZhu` 改为 `mingZhuKey / shenZhuKey`

特别敏感的是 `buildInterpretCards` 的 chip label，当前直接拼接：
```ts
mingTags.push({ label: `借${label}`, tone: 'jade' })  // ❌ "借" 中文
mingTags.push({ label: '清宫', tone: 'plain' })       // ❌ "清宫" 中文
```

需改为：
```ts
mingTags.push({ kind: 'borrowedStar', starKey, brightnessKey })  // ✅ 结构化
mingTags.push({ kind: 'cleanPalace' })                            // ✅ 结构化
```

UI 渲染：
```vue
<span v-if="tag.kind === 'borrowedStar'">
  {{ t('ziwei.tag.borrowedStarFmt', { star: t(`ziwei.starNames.${tag.starKey}`), brightness: t(`ziwei.brightness.${tag.brightnessKey}`) }) }}
</span>
```

### 1.4 UI 组件改造（必改）

7 个组件需要从"消费 string"改为"消费 key + i18n 渲染"：

1. `ZiweiPalace.vue` - star.name → `t(\`ziwei.starNames.${star.starKey}\`)`
2. `InterpretCards.vue` - tag.label 改为结构化条件渲染
3. `DecadalDetail.vue` - m.name → `t(\`ziwei.starNames.${m.starKey}\`)`
4. `ZiweiYear.vue` - m.name → `t(\`ziwei.starNames.${m.starKey}\`)`
5. `SoulPalaceView.vue` - subtitle 中的中文星名拼接
6. `SoulMasterView.vue` - 命主/身主名渲染
7. `SihuaReadingView.vue` - subtitle 的天干字符渲染

### 1.5 i18n 三语补充（必改）

`src/locales/modules/ziwei.{lang}.ts` 三语各加：

| 命名空间 | 数量 | 内容 |
|---------|------|------|
| `ziwei.starNames.{starKey}` | 14 主星 + 12 副星 + 6 煞 = 32 颗 × 3 语 = **96 key** | "紫微" / "Zi-Wei" 等 |
| `ziwei.brightness.{brightnessKey}` | 7 档 × 3 语 = **21 key** | "庙" / "Temple-Brilliant" 等 |
| `ziwei.changsheng12.{key}` | 12 神 × 3 语 = **36 key** | "长生" / "Chang-Sheng" 等 |
| `ziwei.ganzhi.heavenlyStems.{key}` | 10 天干 × 3 语 = **30 key** | "甲" / "Jia" 等 |
| `ziwei.ganzhi.earthlyBranches.{key}` | 12 地支 × 3 语 = **36 key** | "子" / "Zi" 等 |
| `ziwei.tag.cleanPalace` | × 3 语 = **3 key** | "清宫" / "Clean Palace" |
| `ziwei.tag.emptyPalace` | × 3 语 = **3 key** | "空宫" / "Empty Palace" |
| `ziwei.tag.borrowedStarFmt` | × 3 语 = **3 key** | "借{star}{brightness}" / "Borrowed {star} {brightness}" |
| `ziwei.fiveElementsClass.{key}` | 5 局 × 3 语 = **15 key** | "水二局" / "Water Stage 2" |
| **合计新增 i18n key** | — | **~243 key** |

## 3. 渐进式迁移策略

避免"all-in 大爆炸"——分 3 阶段，每阶段都可独立发布：

### Phase A: 数据层 + 类型层（0.5 天）
1. 在 `data/index.ts` 添加 STAR_KEY_TO_DISPLAY 反向映射 + i18n key 命名空间
2. 在 `types/index.ts` 给 Star/DaxianMutagenStar/InterpretCard.tags 加可选 `starKey?: StarKey` 字段（**不删 name，不破坏现有 UI**）
3. 在 `core/ziwei.ts` 5 个派生器同时输出新旧字段（双写）
4. 三语 i18n 补 ~243 key（即使 UI 还没用上）
5. 验证：现有 UI 完全不变，但 ZiweiChart 已含 starKey

### Phase B: UI 渐进切换（0.5 天）
1. 选择 1 个组件（建议 `ZiweiPalace.vue` —— 用户首屏看到的）从 name → starKey
2. en locale Playwright 视觉回归测试通过后，切下一个组件
3. 7 组件全部切完后，name 字段在 UI 层 0 引用

### Phase C: 类型层清理（0.3 天）
1. types 中 `name` 字段标 `@deprecated`
2. core 派生器逐个删除 name 字段
3. 类型检查 + 探针全跑
4. 删除 STAR_KEY_TO_DISPLAY 中的兜底逻辑

## 4. 风险与对策

| 风险 | 对策 |
|------|------|
| iztro 输出新增主星 / 副星，未在 STAR_KEY_TO_DISPLAY 中 | 兜底 starKey='unknown' + 显示原 name；监控脚本 `scripts/ziwei/probe-unknown-stars.mjs` 报警 |
| zh-TW 简繁字符差异（已有 ZH_STAR_TO_KEY 简繁双形）| Phase A 反向映射时基于 STAR_KEY_TO_DISPLAY['zh-TW'] 做完整繁体输出 |
| en 翻译命名风格（Pinyin vs 意译）| Phase A 时与产品对齐：14 主星走 Pinyin（"Zi-Wei"），12 宫名走业内意译（"Life Palace"）—— 已在 §3.5 i18n 表确认 |
| 三方四正 SVG / Sanfang 等组件硬编码中文 | Phase B 时纳入扫描清单 |
| chart cache（`ziweiStore.lastComputed`）反序列化时 starKey 字段缺失 | Phase A 双写时不破坏序列化；Phase C 切换前清一次 store |

## 5. 不在本子项目范围内的事项

- **中文星名 / 命主名 / 五行局的中文显示文案** —— 这些已经是 zh-CN 原生 + zh-TW 简繁双形，不属于"中文混入英文"问题
- **资产文本（soulPalace.text 等长文案）的英文翻译** —— 见 [资产翻译状态清单](./2026-04-28-01-资产翻译状态清单.md)，是独立的内容审校工作
- **iztro 库内部的中文** —— 不可控，由 STAR_KEY_TO_DISPLAY 反向映射屏蔽

## 6. 启动条件 / 优先级

**启动条件**：
- en locale 资产翻译完成进度 ≥ 50%（避免 chip 已英文化但正文还是骨架的尴尬）
- 或：产品决策面向英文用户发布 → 必须先做

**优先级**：🟢 中（M3+ 不阻塞，但是英文用户体验上线前的硬门槛）

**与其他工作的依赖关系**：
- ⏬ **不依赖** zh-TW / en 资产文本翻译（Phase A 完全独立）
- ⏫ **被依赖于** 任何"英文版正式上线"决策
